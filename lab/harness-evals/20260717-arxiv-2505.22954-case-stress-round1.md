Mutation:

  `lab/research/citation-ledger.yaml` carry `fitness_status: needs-review`
  and a boilerplate "Sentence-level citation fitness not audited..." note

Expected:
- Validator may pass; report as workflow friction if semantic fitness is not
  actually audited.

Commands:
```bash
PYTHONDONTWRITEBYTECODE=1 python3 scripts/check-citation-fitness.py
```

Actual:
- As-migrated (184 real, mostly-unique per-citation locators, all
  `needs-review`): **exit 0, `OK citation_fitness`.** The check never
  inspects whether `needs-review`/boilerplate notes reflect a real fitness
  review — it only fires `check_citation_bulk_import_state` when ≥3 *active*
  citations share a byte-identical `(context, locator, section, quote)`
  signature. Forcing 5 duplicate-signature entries at `fitness_status:
  strong` correctly triggered `citation-ledger has repeated strong/adequate
  citation contexts marked as unaudited` (exit 1) — so the guard rail exists,
  but a paper where every citation gets a distinct (even if superficial)
  locator sails through with zero real fitness auditing.

Classification: **documentation friction** — matches the catalog's own
predicted outcome. `check-citation-fitness.py` verifies ledger
*completeness* (every active citation has a bibkey/purpose/context/status)
and catches literal duplicate-boilerplate bulk imports, but does not and
cannot verify that a `needs-review`/`strong` fitness label reflects an
actual read of the source. This is inherent to a structural check, not a
bug — flagging so downstream readers don't mistake `citation_fitness: OK`
for "citations were manually reviewed."

Follow-up: no action on the harness; case debt is tracked via
`fitness_review_status: pending` on the ledger (already recorded).

### P15: Numeric exception masking

Mutation:
- Appended an unrelated, unregistered numeric claim to
  `paper/sections/06_conclusion.tex`: "In a held-out replication, the
  discovered agent additionally solved 2025 previously-unseen tasks." No
  ledger change. This exploits the already-committed
  `state/numbers/exceptions.yaml` entry `{"pattern": "2025", "match_scope":
  "literal", "reason": "publication/writing date reference..."}`, added
  during this migration to mask real date mentions (arXiv writing dates,
  "Spring 2024", etc.) in `paper/sections/10_appendix.tex`.

Expected:
- Numeric check should fail unless the exception is explicitly scoped.

Commands:
```bash
PYTHONDONTWRITEBYTECODE=1 python3 scripts/check-numeric-consistency.py
```

Actual:
- **exit 0, `OK numeric_consistency`.** The fabricated "2025 previously-unseen
  tasks" literal is silently masked because the exception matches by literal
  string value only (`match_scope: literal`), with no `path_pattern`
  restricting it to the specific files/lines where the real date mentions
  live. `paper/refs.bib`'s harness does support `path_pattern` (regex
  against relpath) for exactly this kind of scoping — it's used elsewhere in
  this same file for the `paper/supplementary/**` bulk-content exception —
  but the three date exceptions (`2024`, `2025`, `2026`) were authored
  without it.

Classification: **case ledger debt** (self-inflicted: the exception
mechanism supports tighter scoping and this migration didn't use it for the
date exceptions). Not an upstream harness gap — the fix is local.

Follow-up: tighten `state/numbers/exceptions.yaml`'s `2024`/`2025`/`2026`
entries with `path_pattern` scoped to the specific files where those literal
years legitimately appear (`paper/sections/10_appendix.tex`,
`state/conference-template.yaml`-adjacent venue metadata), so a fabricated
number that happens to equal a masked year elsewhere is not accidentally
exempted. Also worth an upstream note: `match_scope: literal` with no
`path_pattern` is a foot-gun for exactly this scenario — the catalog's own
worked example (`state/numbers/exceptions.yaml`'s seeded `"2027"` entry in
the pristine template) has the same unscoped shape, so this is a pattern the
template itself invites.

## Missed-by-both-leaf-and-profile findings (headline)

Only P15 is a clean "leaf check misses it, and it's fixable purely at the
case-ledger level" finding — the profile was unavailable to cross-check, but
the mechanism (`path_pattern`) to fix it exists in the leaf check itself, so
this is squarely case ledger debt, not an upstream proposal.

P9, P20, and P22 are all "the gate exists and works when properly
configured/populated, but the shipped default (empty citation-fitness
enforcement threshold, absent `physical_validation`, `raw_template: TODO`)
leaves it inert" — these read as **documentation friction** /
